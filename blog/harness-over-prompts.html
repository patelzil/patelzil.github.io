Agents Harness Engineering Developer Workflow

The Harness Is Doing More Work Than You Think

April 21, 2026 3 min

Our dev environment spans multiple repositories, each with its own startup sequence: branch sync, UI rebuild, service start, sometimes a non-standard path like a nested adapter. A full refresh was more than thirty minutes of manual steps, so the obvious move was to hand it to an agent: pull and start all the apps, and move on.

That didn't work reliably. The failures were not about reasoning; the agent could clearly follow instructions. The problem was that for every invocation, it was silently re-answering questions I had never written down.

Why better prompts didn't help

My first instinct was that the prompt needed to be more specific, so I wrote a longer one. Then a longer one. It kept failing in new ways. Eventually I sat down and listed the things the agent was having to infer on every run: which repos count as "all apps", which branch to use in each repo, what to do when a branch exists in some repos but not others, which startup script each app uses, whether UI rebuilds are already baked into those scripts, and how to handle the nested adapter in one repo.

Each of those is a judgement call that does not appear in the code and does not appear in the prompt. A longer prompt made the intent clearer without making any of those answers more stable. Inconsistency and retries were the predictable result.

What the runbook encodes

So I stopped tuning the prompt and started building around it. I wrote a small, reusable multi-app runbook that prompts once for a target branch, defaults to running every app, checks each repo independently for that branch, and falls back to develop where the branch does not exist. None of this is clever; it is mostly just writing down what I would have done manually.

The agent now runs inside a defined execution path instead of reconstructing one from context on every invocation. Setup went from a long manual process to a single guided workflow, and the failure modes became specific and fixable instead of vaguely vibes-y.

Smart shell vs. playbook

The distinction that turned out to matter, more than I expected, is agent-as-smart-shell versus agent-running-a-playbook. When the agent is a smart shell, you are relying on the model to rediscover your repo layout, your conventions, and your exception handling on every call. When the agent is running a playbook, the domain knowledge lives in the system around the model, and the model is free to do the part it is actually good at: the judgement inside each well-defined step.

The LangChain team recently reported that the same model, wrapped in a tighter harness, jumped from roughly Top 30 to Top 5 on Terminal Bench 2.0, 52.8% to 66.5%. Their writeup is worth reading in full. It lines up with what I keep seeing in my own work: the harness is doing more work than people assume, and the returns on tightening it up are larger than the returns on a better prompt or a fancier model.

A prompt describes intent. A runbook makes execution repeatable. If your agent keeps underperforming, the more useful question probably isn't which model to switch to. It's what the agent is still having to infer that you should have just written down.
